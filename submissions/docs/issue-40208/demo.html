🐛 Unprefixed Selector Names in Component CSS

Issue #40208: Naming Convention Inconsistency Expands Public API

The Problem

❌ Current Behavior (Mixed Prefixes)

The Issue: Component CSS files ship with inconsistent selector naming conventions.

Expected: All selectors should use `ease-` prefix (e.g., `.ease-badge`)

Actual: Some use `em-` prefix (e.g., `.em-badge`), some unprefixed, some context-dependent

The Risk: Expands public API surface, increases naming collision risk, creates confusion for developers

⚠️ Selector Naming Violations Found

components/badges.css

.em-badge ❌ Wrong Prefix
.em-badge-danger ❌ Wrong Prefix
.em-badge-success ❌ Wrong Prefix

components/breadcrumb.css

.bg-dark .ease-breadcrumb-active ⚠️ Mixed Context
.breadcrumb-divider ❌ Unprefixed

components/skeleton.css

.skeleton-card-header .ease-skeleton-text ⚠️ Mixed Context
.skeleton-loading ❌ Unprefixed

🔴 Impact & Risk Assessment

Public API Pollution

Unprefixed selectors become part of framework's public API, hard to deprecate later

Naming Collisions

Users' CSS might collide with `.em-badge`, `.breadcrumb-divider`, etc.

Inconsistency

Developers expect `ease-` prefix throughout, but encounter `em-` and unprefixed selectors

Maintenance Debt

Longer refactoring period required to migrate legacy selectors

Root Cause Analysis

Why Do Unprefixed Selectors Exist?

1. Legacy Component Contributions

Components added before naming convention was finalized still use old prefixes (`em-`) or no prefix.

Contributors followed different standards during development.

2. No Validation During Submission

There's no automated check to enforce selector naming conventions in pull requests.

Maintainers review manually, but inconsistencies slip through.

3. Context-Based Selectors

Some selectors rely on parent context (`.bg-dark .ease-breadcrumb-active`) mixing prefixed and unprefixed.

Intent unclear: Is `.bg-dark` a framework selector or external context?

4. No Scanning Tool

No automated tool scans CSS files to detect and report naming violations.

Solutions

Approach 1: Fix Components Immediately

Manual Remediation

✅ Pros

  • Direct fix to existing files
  • Simple find-and-replace
  • Immediate consistency

❌ Cons

  • Breaking change for users
  • Requires migration guide
  • One-time effort only

Steps:

# In components/badges.css - .em-badge { } + .ease-badge { } - .em-badge-danger { } + .ease-badge-danger { } # In components/breadcrumb.css - .breadcrumb-divider { } + .ease-breadcrumb-divider { } # In components/skeleton.css - .skeleton-loading { } + .ease-skeleton-loading { }

Approach 2: Add Validation

Automated Prevention

✅ Pros

  • Prevents future violations
  • CI/CD integration
  • Immediate feedback

❌ Cons

  • Requires setup effort
  • Tool configuration needed
  • Doesn't fix existing issues

Validate Selectors Script:

// scripts/validate-selectors.mjs import fs from 'fs'; import glob from 'glob'; const SELECTOR_PATTERN = /^\.([a-z-]+)/gm; const REQUIRED_PREFIX = 'ease-'; function validateSelectors(componentDir) { const files = glob.sync(`${componentDir}/**/*.css`); const violations = []; files.forEach(file => { const content = fs.readFileSync(file, 'utf8'); const matches = [...content.matchAll(SELECTOR_PATTERN)]; matches.forEach(match => { const selectorName = match[1]; if (!selectorName.startsWith(REQUIRED_PREFIX)) { violations.push({ file, selector: `.${selectorName}`, line: content.substring(0, match.index) .split('\n').length }); } }); }); return violations; } const violations = validateSelectors('components'); if (violations.length > 0) { console.error('❌ Selector naming violations found:'); violations.forEach(v => { console.error( ` ${v.file}:${v.line} - ${v.selector}` ); }); process.exit(1); } console.log('✅ All selectors properly prefixed');

🎯 Recommended Solution

Use Both Approaches Together:

  1. Phase 1 (Immediate): Fix existing components to use `ease-` prefix
  2. Phase 2 (Prevention): Add `validate:selectors` script to CI/CD
  3. Phase 3 (Documentation): Document naming convention for future contributors
  4. Phase 4 (Deprecation): If needed, support legacy aliases with warnings

Why Both? Fixes existing violations immediately while preventing future ones automatically.

Naming Convention Standards

✅ Correct Naming Patterns

Component Base Classes

.ease-badge .ease-breadcrumb .ease-skeleton

Component Modifiers

.ease-badge-danger .ease-badge-success .ease-breadcrumb-active

Component Child Elements

.ease-breadcrumb-divider .ease-skeleton-text .ease-badge-icon

Component States

.ease-breadcrumb-disabled .ease-skeleton-loading .ease-badge-outlined

❌ Anti-Patterns to Avoid

Legacy em- Prefix

.em-badge .em-breadcrumb Use ease- instead

Unprefixed Selectors

.badge .breadcrumb Add ease- prefix

External Context Mixing

.bg-dark .ease-breadcrumb Only framework selectors in components

Inconsistent Nesting

.skeleton-card .ease-skeleton Keep all selectors prefixed

Technical Details

Prefix Strategy

ease- is the official namespace prefix for all framework selectors

Reduces collision risk, improves discoverability, clarifies ownership

Selector Scope

Only framework-owned selectors require prefix

External utility selectors (e.g., from utilities.css) may differ

Migration Path

Breaking change - requires major version bump

Provide migration guide for v2.0 release

Backward Compatibility

Option: Keep em- aliases for backward compat

Document deprecation, encourage migration
