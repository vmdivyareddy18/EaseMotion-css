01 · Rationale

Why Motion Decisions Matter

Picking an animation pattern deliberately, instead of by habit, has a direct and compounding effect on how a product feels and how easy it is to maintain.

UX

The right motion pattern communicates what changed and where attention should go, without asking the user to guess.

Accessibility

A structured decision process makes it natural to consider reduced motion and cognitive load from the start, not as a retrofit.

Maintainability

Deliberate choices are easier to document and justify later than motion that was tuned by feel and never explained.

Consistency

Mapping scenarios to patterns keeps similar interactions animating the same way across an entire product surface.

Performance

Choosing a pattern with performance in mind up front avoids expensive rewrites once jank shows up in production.

02 · Framework

Motion Selection Criteria

Before reaching for a pattern, weigh the interaction against these eight criteria. Together they narrow down what motion actually needs to do.

User Intent

What is the person trying to accomplish, and does the motion support that goal directly?

Context Preservation

Does the transition keep the user oriented relative to where they came from?

Information Hierarchy

Does the motion draw attention to what matters most, in the right order?

Interaction Frequency

Frequently repeated interactions need shorter, quieter motion than rare ones.

Accessibility

Can the pattern be safely reduced or removed for users who need less motion?

Device Constraints

Will the pattern hold up on lower-powered devices and smaller viewports?

Performance

Does the implementation rely on cheap, GPU-friendly properties, or expensive layout changes?

Cognitive Load

Does the motion reduce the effort needed to understand what happened, or add to it?

04 · Patterns

Pattern Comparison

Each pattern trades off differently across clarity, cost, and complexity. Use this comparison to pick the pattern that fits the scenario, not the trend.

Fade

Purpose
Introduce or remove content with minimal spatial implication.
Strengths
Simple, cheap, and rarely distracting.
Limitations
Communicates almost nothing about direction or origin.
Best Use Cases
Tooltips, loading states, subtle content swaps.

Slide

Purpose
Show that content is entering from or exiting toward an edge.
Strengths
Clear directional cue, familiar to users.
Limitations
Can feel heavy if overused for small, frequent changes.
Best Use Cases
Drawers, toasts, notification panels.

Scale

Purpose
Emphasize an element growing into or shrinking out of focus.
Strengths
Reinforces a sense of origin, works well from a trigger point.
Limitations
Overuse can feel bouncy or attention-seeking.
Best Use Cases
Modals, dropdowns, popovers.

Stagger

Purpose
Introduce a group of related items in a readable sequence.
Strengths
Reduces visual overwhelm from many items appearing at once.
Limitations
Adds perceived latency if delays are too long.
Best Use Cases
Dashboard widgets, lists, grids of cards.

Sequential

Purpose
Move through an ordered series of items one at a time.
Strengths
Matches the mental model of stepping through a sequence.
Limitations
Not suited to unordered or unrelated content.
Best Use Cases
Galleries, carousels, step-based flows.

Progressive Reveal

Purpose
Expose additional detail gradually without losing prior context.
Strengths
Keeps content discoverable without overwhelming the initial view.
Limitations
Can feel slow if content is expected to appear instantly.
Best Use Cases
Accordions, expandable detail panels, onboarding.

Shared Axis

Purpose
Signal forward or backward movement along a spatial or temporal axis.
Strengths
Strong sense of direction between related views.
Limitations
Requires a consistent axis convention to stay legible.
Best Use Cases
Page navigation, wizard steps, tabbed views.

Container Transform

Purpose
Morph one element directly into its expanded or detailed form.
Strengths
Preserves visual continuity between summary and detail states.
Limitations
More complex to implement and can be costly if not transform-based.
Best Use Cases
Card expansion, list-to-detail transitions.

06 · Accessibility

Accessibility

Every pattern in this matrix should be evaluated against these four accessibility considerations before it ships.

prefers-reduced-motion

Provide a reduced or removed alternative for any non-essential motion, honoring the user's system preference.

Motion Sensitivity

Avoid large-scale movement, parallax, or spinning effects that can cause discomfort for sensitive users.

Focus Visibility

Keep focus indicators visible and correctly positioned throughout every transition, not just at rest.

Reduced Cognitive Load

Favor motion that clarifies what changed over motion that is simply decorative or attention-grabbing.

07 · Performance

Performance

A pattern is only a good choice if it also performs well across the devices your users actually have.

GPU-Friendly Transforms

Prefer transform and opacity, which the browser can animate on the compositor without recalculating layout.

Opacity

Opacity changes are inexpensive and combine well with transform for smooth, low-cost transitions.

Avoiding Layout Shift

Animating width, height, or margin directly can trigger repeated layout recalculation and visible jank.

Efficient Transitions

Keep the number of simultaneously animating properties small, and reuse the same duration and easing tokens.

08 · Best Practices

Best Practices

A short set of habits that make the decision matrix easier to apply consistently.

Choose Purpose Before Animation

Decide what the motion needs to communicate before picking how it should look.

Reuse Existing Motion Patterns

Check the matrix for a similar scenario before inventing a new pattern.

Keep Transitions Predictable

Similar interactions should animate the same way everywhere they appear.

Respect Accessibility

Treat reduced motion and focus visibility as requirements, not optional polish.

Test Motion Consistently

Revisit chosen patterns whenever a component or its context changes meaningfully.

Motion is a decision, not a default. Use this matrix as a starting point, weigh each scenario against the selection criteria, and choose the pattern that best serves clarity, accessibility, and performance together.
