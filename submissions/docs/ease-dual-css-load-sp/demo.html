Dual Stylesheet Cascade Surprise

Beginners often link both easemotion.css and easemotion.min.css “just to be safe.” That doubles the framework in the cascade. Toggle modes below and watch the surprise.

Issue #46185
Debug vs prod rule: Use one EaseMotion stylesheet at a time — readable source while debugging, minified bundle in production. Never ship both.

What goes wrong when both load

  • Doubled download Bytes

    You fetch roughly two full copies of EaseMotion (~178 KB raw min plus the readable tree of @imports). Users pay twice; nothing new ships.

  • Doubled rules in the cascade Cascade

    Every .ease-* rule appears twice. Equal specificity → the later stylesheet wins. That “last wins” surprise looks like a random bug when you insert custom CSS between the two links.

  • Overrides get stomped Debugging

    A tweak after easemotion.css but before .min.css can disappear. DevTools looks haunted; the dual load is the culprit.

  • Harder mental model DX

    Source exists for reading. Min exists for shipping. Mixing them fights the “one composition” idea of a curated framework.

Debug vs production

Goal Use Don’t
Read / debug class source easemotion.css Also load .min.css
Ship to production / CDN demo easemotion.min.css Also load readable entry
“Just to be safe” Pick one intentionally Link both — never safer

Recommendation

  • ✓ Production: one easemotion.min.css (CDN or npm).
  • ✓ Local debugging: one easemotion.css when you need readable imports.
  • ✓ Put custom overrides after your single EaseMotion link.
  • ✕ Do not load source and min together on the same page.
  • ✕ Do not blame EaseMotion first when dual-load stomps your CSS.
