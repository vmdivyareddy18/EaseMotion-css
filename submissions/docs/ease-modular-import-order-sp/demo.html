Modular Import Order Violation Demo

README Option 4 lets you load only the animation modules you need — but easemotion/variables.css must come first. This lab shows broken vs working speeds/easings side by side, with copy-paste correct snippets.

Issue #46175
Rule: easemotion/variables.css must always load before modular animation files (fade.css, slide.css, …). Those modules use tokens like --ease-speed-medium and --ease-ease. Without them, the animation shorthand becomes invalid and the fade never looks right.

Why variables must load first

  • Tokens power the utilities Tokens

    fade.css declares animation: ease-kf-fade-in var(--ease-speed-medium) var(--ease-ease) both. Those custom properties live in variables.css.

  • Missing vars invalidate the shorthand Broken

    If --ease-speed-medium / --ease-ease are undefined, the whole animation declaration becomes invalid — so you get no (or wrong) duration and easing, even though the keyframes file loaded.

  • Option 4 is modular, not magic Docs

    Loading only fade.css is fine for tree-shaking size — but every modular setup still needs variables.css once, at the top of the list.

Recommended pattern

Always start modular imports with variables, then add only the animation categories you need — same order as README Option 4.

  • ✓ Put easemotion/variables.css first.
  • ✓ Then add fade.css, slide.css, etc.
  • ✓ Replay both panels above to compare computed duration/easing.
  • ✕ Do not ship animation modules without the variables sheet.
  • ✕ Do not assume a blank fade means EaseMotion keyframes are “broken.”
