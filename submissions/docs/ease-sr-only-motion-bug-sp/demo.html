ease-sr-only + Animated Parent Becomes Announced Twice

Visually hidden labels inside motion-animated regions can be read twice by screen readers. Learn the unsafe pattern, why it happens, and copy-ready fixes.

Short answer: Do not put ease-sr-only text inside a looping / transform-animated parent, and do not duplicate the same string with both aria-label and an SR-only child. Keep motion on a decorative sibling (aria-hidden="true") and keep the accessible name on a stable control.

What ease-sr-only does

From core/utilities.css: the class clips the element out of the visual layout but leaves it in the accessibility tree so assistive tech can still read it. 

.ease-sr-only {
  position: absolute !important;
  width: 1px !important;
  height: 1px !important;
  overflow: hidden !important;
  clip: rect(0, 0, 0, 0) !important;
  clip-path: inset(50%) !important;
  /* …plus margin / padding / border resets */
}

Why animated parents cause double announcement

  1. Duplicate accessible names. aria-label="Notifications" plus a child <span class="ease-sr-only">Notifications</span> can cause some SR / browser combinations to expose the name twice when focus or activation updates.
  2. Motion on the naming container. Transforms / opacity animations on the ancestor that owns the name can trigger reflow or tree updates; live regions and some SR modes re-speak content that “changed.”
  3. Visible text + SR-only synonym. A chip that shows “Online” and also hides “Status: online, connection active” inside the same animated node often announces both strings.
  4. Looping classes amplify the bug. Infinite pulses (ease-pulse, ease-ping) make accidental re-announcements more noticeable during testing.

Pattern comparison

Pattern What SR hears Verdict
Animated button + aria-label + nested ease-sr-only (same string) “Notifications… Notifications” Unsafe
Animated wrapper around visible text + extra SR-only status phrase “Online… Status: online, connection active” Unsafe
Stable button with one aria-label; motion on aria-hidden sibling “Notifications, button” Safe
Stable text node; animate decorative indicator only “Online” (once) Safe
Icon-only button with only ease-sr-only (no duplicate aria-label), no parent animation Single label from SR-only child Safe

Accessibility checklist

  • Use one naming strategy: either aria-label or ease-sr-only — not both with the same text.
  • Keep EaseMotion classes on decorative children marked aria-hidden="true", not on the labeled control when possible.
  • Avoid nesting SR-only “extra context” inside looping animated parents; put context in visible text or a separate static note.
  • Test with a real screen reader (NVDA / VoiceOver / Narrator) after adding motion — simulators only teach the concept.
  • Respect prefers-reduced-motion for loops; less motion means fewer accidental re-announcements during focus.

Copy-ready snippets

Paste these into your markup. Prefer the safe pattern for production UI.

❌ Unsafe — duplicate name + animated parent
Copied 
<button class="ease-pulse" aria-label="Notifications">
  🔔
  <span class="ease-sr-only">Notifications</span>
</button>
✅ Safe — one name, motion on decorative sibling
Copied 
<button aria-label="Notifications">
  <span class="ease-pulse" aria-hidden="true"></span>
  <span aria-hidden="true">🔔</span>
</button>
✅ Safe — SR-only only (no aria-label), no parent motion
Copied 
<button type="button">
  <span aria-hidden="true">🔍</span>
  <span class="ease-sr-only">Search</span>
</button>
✅ Safe — status chip with decorative pulse
Copied 
<div role="status">
  <span class="ease-pulse" aria-hidden="true"></span>
  Online
</div>
