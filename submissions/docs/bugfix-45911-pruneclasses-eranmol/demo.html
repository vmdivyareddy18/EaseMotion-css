Bug Fix #45911 — pruneClasses nested CSS

Demonstrates that .ease-fade-in (with a nested &:hover block) and .ease-fade-in-slow both survive tree-shaking when both classes are used.

ease-fade-in

This card uses .ease-fade-in, which contains a nested &:hover rule. Before the fix, the nested brace caused the regex to break and silently drop the next class.

✓ Preserved by pruneClasses

ease-fade-in-slow

This card uses .ease-fade-in-slow. Before the fix, this class was silently dropped from the pruned CSS output — even though it was listed in usedClasses.

✓ No longer dropped (issue fixed)

What was the bug?

/* This CSS caused pruneClasses to fail */
.ease-fade-in {
  animation: ease-kf-fade-in 0.3s ease-out;
  &:hover { animation-play-state: paused; }  /* ← nested braces */
}

/* This rule was silently dropped from output */
.ease-fade-in-slow {
  animation: ease-kf-fade-in 0.6s ease-out;
}

Root cause

The pruneClasses function in engine/optimizer.js used the regex pattern [^{}]* to match a rule body. This pattern cannot handle nested braces — it stops at the first inner closing }, leaving the cursor at the wrong position in the CSS string and causing the next rule to be skipped entirely.

The fix

Replaced the naive regex with a brace-counting parser that walks forward character by character, incrementing depth on { and decrementing on }. The rule body ends only when depth returns to zero — correctly handling any level of nesting.
