Common Motion Incidents

These are the symptoms most frequently reported in production. Use them as an entry point, then move to Root Cause Analysis to narrow down the actual defect.

Animation Jank

Symptoms
Choppy or uneven motion, dropped frames
Possible Causes
Layout-triggering properties, too many concurrent animations
Impact
Perceived low quality, especially on lower-end devices

Layout Shift

Symptoms
Nearby content jumps or reflows during animation
Possible Causes
Animating width, height, or margin directly
Impact
Poor CLS scores, disorienting user experience

Flickering

Symptoms
Brief flash or incorrect state at animation boundaries
Possible Causes
Missing animation-fill-mode, specificity conflicts
Impact
Feels broken or unpolished even if functionally correct

Unexpected Restart

Symptoms
Animation replays from the start unexpectedly
Possible Causes
Element remounting, class toggling on every render
Impact
Distracting, breaks continuity of the interface

Battery Drain

Symptoms
Unusually high power usage reported on animation-heavy pages
Possible Causes
Continuous animations running while tab is hidden
Impact
Degrades mobile experience, user complaints

Repaint Issues

Symptoms
Visible smearing, ghosting, or slow-to-update regions
Possible Causes
Animating box-shadow, filter, or background-position
Impact
Reduced frame rate, especially on large elements

Reduced Motion Problems

Symptoms
Some animations respect reduced motion settings, others don't
Possible Causes
Per-component overrides instead of centralized handling
Impact
Accessibility risk for motion-sensitive users

Performance Diagnostics

Layout Thrashing

Repeatedly reading a layout property and then writing a style change forces the browser to recalculate layout multiple times per frame. Batch all reads before all writes to avoid this.

Paint-Heavy Animations

Properties like box-shadow, filter, and background-position force a repaint every frame. Prefer compositor-only properties wherever the visual result allows it.

GPU-Friendly Properties

Only transform and opacity can animate purely on the compositor thread, skipping layout and paint entirely. Treat these as the default choice for motion.

Transform vs Width

Animating width recalculates layout on every frame. Animating transform: scale() achieves a similar visual effect while staying on the compositor thread.

Opacity vs Display

Toggling display removes an element from the render tree instantly, which cannot be transitioned. Animate opacity (and visibility, where needed) instead.

Accessibility Troubleshooting

prefers-reduced-motion

Verify every animated component individually with the OS-level reduced motion setting enabled — support is often implemented inconsistently across a codebase.

Focus Visibility

Animated transforms on a focused element can misalign or hide the focus ring mid-transition. Keep focus indicators on a stable, non-animated layer.

Vestibular Considerations

Large parallax, rapid scaling, and spinning motion can cause discomfort independent of general reduced-motion preference. Reduce magnitude, not just duration.

Keyboard Users

Confirm that animated state changes never trap or skip keyboard focus, and that interactive elements remain operable throughout a transition.

Screen Readers

Synchronize aria-live announcements with the meaningful point of an animation, rather than firing before the visual change has actually completed.

Browser Compatibility

Chrome

Broadest support for modern motion features such as scroll-driven animations and View Transitions, but frequently ships experimental behavior ahead of other engines.

Firefox

Generally strong standards compliance; newer experimental CSS features often land later than in Chromium-based browsers.

Safari

Historically slower to adopt new animation APIs; pay particular attention to backdrop-filter, clip-path, and compositing differences.

Edge

Chromium-based and generally aligned with Chrome, but still worth verifying independently for enterprise-managed policy differences.

Debugging Workflow

A repeatable sequence for investigating any motion-related production issue.

  1. 1

    Identify Problem

    Reproduce reliably across device, browser, and settings.

  2. 2

    Inspect Animation

    Isolate the element and review its computed styles and timing.

  3. 3

    Determine Cause

    Profile with DevTools rather than guessing at the source.

  4. 4

    Apply Fix

    Change one variable at a time to confirm the actual cause.

  5. 5

    Retest

    Verify across target browsers, real devices, and reduced motion.

  6. 6

    Ship

    Document the root cause and fix for future reference.

Production Checklist

  • Timing consistency across related animations
  • Easing consistency across related animations
  • Accessibility verified, including reduced motion
  • Performance profiled on a real, non-flagship device
  • Browser compatibility confirmed across targets
  • Responsive behavior tested at multiple breakpoints
  • Reduced-motion support implemented and manually verified

Best Practices

Tip

Default to transform and opacity for any new animation.

Warning

Avoid overusing will-change — remove it once the animation completes.

Tip

Centralize prefers-reduced-motion handling instead of per-component overrides.

Note

Profile before optimizing — assumptions about performance are frequently wrong.
