Animation Performance Best Practices

A developer guide to writing smooth, efficient CSS animations — the same principles EaseMotion CSS is built on.

1. Use transform and opacity — the compositor-only pair

Only transform and opacity can animate entirely on the GPU compositor thread, bypassing the browser's layout and paint steps. Every other property (left, top, width, height, background-color…) forces the main thread to recalculate layout or repaint pixels on every frame.

Property Triggers Layout? Triggers Paint? GPU Compositor? Verdict
transform NoNo✅ Yes Use this
opacity NoNo✅ Yes Use this
left / top / right / bottom ✗ Yes✗ YesNo Avoid in animations
width / height ✗ Yes✗ YesNo Avoid in animations
background-color No✗ YesNo Avoid in keyframes
box-shadow No✗ YesNo Use sparingly

✅ Good — transform: translateX()

Slides in with transform
/* ✅ Compositor-only — no layout, no paint */
@keyframes slide-in {
  from { opacity: 0; transform: translateX(-40px); }
  to   { opacity: 1; transform: translateX(0);      }
}
.slide-in { animation: slide-in 600ms ease both; }

❌ Bad — left (layout-triggering)

Slides in with left
/* ❌ Triggers layout recalc every frame — jank risk */
@keyframes slide-in-bad {
  from { opacity: 0; left: -40px; }
  to   { opacity: 1; left: 0;      }
}
.slide-in-bad { position: relative; animation: slide-in-bad 600ms ease both; }

2. Fading? Use opacity — not visibility or display

opacity transitions on the compositor. visibility and display are not animatable — they snap instantly. Use opacity combined with pointer-events: none when you need to hide and show elements.

✅ opacity fade-in

Fades in smoothly
@keyframes fade-in {
  from { opacity: 0; }
  to   { opacity: 1; }
}
.fade-in { animation: fade-in 500ms ease both; }

3. Progress bars — scaleX, not width

Animating width from 0 → 100% forces the browser to recalculate layout on every frame. Use transform: scaleX() with transform-origin: left center instead — visually identical, compositor-only.

✅ scaleX (compositor-only)

/* ✅ No layout recalc */
.bar { transform-origin: left center; }
@keyframes progress { from { transform: scaleX(0); } to { transform: scaleX(1); } }

❌ width animation (layout every frame)

/* ❌ Layout recalc on every frame */
@keyframes progress-bad { from { width: 0; } to { width: 100%; } }

4. will-change — hint early, scope tightly

will-change tells the browser to promote the element to its own GPU layer before the animation begins, eliminating the first-frame promotion cost. Apply it only when needed and only for the properties that will actually change. Never use will-change: all.

✅ will-change scoped to :hover

Hover me — GPU layer on demand
/* ✅ Layer only active while hovering */
.card { transition: transform 300ms ease, box-shadow 300ms ease; }
.card:hover {
  will-change: transform, box-shadow;
  transform: translateY(-6px) scale(1.03);
}

/* ❌ Never do this — wastes GPU memory for the entire page lifetime */
/* .card { will-change: all; } */

5. GPU layer promotion with translateZ(0)

A null 3D transform forces the browser to create a compositor layer. Use it as a last resort when an element still jitters despite using transform. Overusing it exhausts GPU memory and hurts performance on low-end devices.

💡 Use sparingly

Hover me — explicit GPU layer
/* Forces its own compositor layer from the start */
.element {
  transform: translateZ(0);   /* or translate3d(0,0,0) */
  will-change: transform;
}

/* Only if you observe jank after using transform + opacity */

6. Always respect prefers-reduced-motion

Users with vestibular disorders or motion sensitivity can enable "Reduce motion" in their OS. EaseMotion CSS handles this globally in core/animations.css. Your submissions must do the same.

✅ Accessible animation with motion guard

Bouncing — stops if reduce-motion is on

Enable "Reduce motion" in your OS Accessibility settings to see the animation stop. 

@keyframes bounce {
  0%, 100% { transform: translateY(0); }
  50%       { transform: translateY(-14px); }
}

.animated { animation: bounce 1s ease-in-out infinite; }

/* Always pair with a reduced-motion override */
@media (prefers-reduced-motion: reduce) {
  .animated { animation: none; }
}

7. CSS containment for independent subtrees

contain: layout style paint tells the browser that changes inside this element cannot affect anything outside it. This narrows the browser's scope for recalculation, especially useful in list items, cards, or repeating animated components.

✅ contain scopes browser recalculation

Contained element — hover me
/* Scopes layout/paint recalculation to this element */
.card {
  contain: layout style paint;
}

/* For elements that also don't overflow their bounds: */
/* contain: strict; */

Performance Cheatsheet

  • Animate only transform and opacity in keyframes
  • Replace left/top with translateX/translateY
  • Replace width animations with scaleX + transform-origin
  • Apply will-change only in :hover / :focus — not globally
  • Use translateZ(0) only as a last resort for stubborn jank
  • Always add a @media (prefers-reduced-motion: reduce) override
  • Use contain: layout style paint on independent animated components
  • Prefer transition over @keyframes for simple state changes
