🐛 Case-Sensitivity Collision Fix

EaseMotion CSS Issue #38298

❌ The Problem

Issue: Git clone fails on Windows and macOS due to case-sensitivity collision

When cloning the EaseMotion CSS repository on a case-insensitive file system (Windows or macOS), Git detects a collision between two folder names that differ only in capitalization:

submissions/examples/ ├── Tada-Click-animation/ ← uppercase "C" └── Tada-click-animation/ ← lowercase "c"

This causes Git to throw a warning and fail to properly checkout all files:

warning: the following paths have collided (e.g. case-sensitive paths on a case-insensitive filesystem) and only one from the same colliding group is in the working tree: 'submissions/examples/Tada-Click-animation/demo.html' 'submissions/examples/Tada-click-animation/demo.html'
💡 Why it happens: Linux/Unix filesystems are case-sensitive (Tada-Click-animation ≠ Tada-click-animation), but Windows and macOS are case-insensitive (they treat them as the same). Git tracks both as separate paths, creating a collision that breaks on case-insensitive systems.

⚠️ Impact

Who is affected:

  • All Windows users cloning the repository
  • All macOS users cloning the repository
  • CI/CD pipelines on Windows runners
  • New contributors trying to set up locally

Consequences:

  • Incomplete repository clone
  • Missing or corrupted files from one folder
  • Git shows "untracked" or "conflicting" status
  • Difficulty resolving the issue without understanding Git's case handling

✅ The Solution

Fix: Keep the lowercase folder and remove the uppercase folder from Git tracking

Before (Broken)

submissions/examples/
Tada-Click-animation/
Tada-click-animation/

After (Fixed)

submissions/examples/
✓ Tada-click-animation/
Why lowercase? Lowercase naming is a Git best practice and matches EaseMotion's existing naming convention (e.g., ripple-toggle-switch, cascading-barcode). It's also more compatible with URLs and command-line tools.

🔧 Why This Is A Bug Fix (Not A Feature Deletion)

Aspect Normal Feature Deletion (Not Allowed) This Fix (Structural Bug Fix)
Scope Removing user-facing feature or demo Fixing a Git structural issue
Content Deleting animation or component code Two identical folders (content is duplicate)
Impact User loses access to a feature All users gain proper access
Status Breaks functionality Fixes functionality
Key Point: The duplicate uppercase folder doesn't contain unique content—it's a duplicate entry in Git's tracking. Removing it from Git (while keeping the lowercase version) is a structural fix that benefits all users, not a destructive deletion.

🛠️ How To Implement The Fix

  1. In Git, remove the uppercase folder from tracking:
    git rm -r --cached "submissions/examples/Tada-Click-animation"
  2. Verify only the lowercase folder remains:
    git ls-files submissions/examples/ | grep -i tada # Expected output: # submissions/examples/Tada-click-animation/demo.html # submissions/examples/Tada-click-animation/style.css # submissions/examples/Tada-click-animation/README.md
  3. Commit the fix:
    git add -A git commit -m "fix: resolve case-sensitivity collision in submissions/examples (#38298) - Remove duplicate uppercase folder 'Tada-Click-animation' from Git tracking - Keep lowercase folder 'Tada-click-animation' (follows naming convention) - Fixes git clone failures on Windows and macOS - Resolves Issue #38298"
  4. Push to remote:
    git push origin main
Note: The git rm --cached command removes the folder from Git tracking only; it doesn't delete files from the working directory (though they don't exist locally anyway on case-insensitive systems).

✔️ Testing The Fix

After the fix is merged, verify on Windows and macOS:

  1. Clone the repository fresh:
    git clone https://github.com/SAPTARSHI-coder/EaseMotion-css.git
  2. Check for Git warnings:
    # No warning should appear about case-sensitive path collision
  3. Verify the folder exists:
    ls submissions/examples/ | grep -i tada # Should output: Tada-click-animation # NOT both Tada-Click-animation and Tada-click-animation
  4. Confirm Git status is clean:
    git status # Expected: "nothing to commit, working tree clean"
