Astro Island Hydration + EaseMotion Class Timing

EaseMotion classes are CSS-first — they work before JavaScript hydrates. But Astro client:* directives control when island JS runs. This guide explains what works immediately vs what waits for hydration.

Key insight: ease-fade-in, ease-hover-grow, and other pure CSS classes apply on first paint — no hydration needed. Only JS-driven animation triggers (adding classes via useEffect, click handlers, etc.) wait for the island to hydrate.

CSS-first vs hydration timeline

What happens when an Astro page with an island loads:

1. Server renders HTML + CSS

Astro sends static HTML with class="ease-fade-in" already in markup. EaseMotion CSS is loaded globally in Layout.astro. Entrance animations start immediately — no JS required.

2. Browser paints (First Contentful Paint)

Pure CSS classes (ease-fade-in, ease-hover-grow, ease-slide-up) are active. Hover effects work via CSS :hover pseudo-class — also no JS needed.

3. Island hydration begins (client:* schedule)

Astro downloads and executes island JavaScript based on the client:* directive. Until this completes, JS event handlers and React/Vue/Svelte state are inactive.

4. Island is interactive

JS-driven behaviors activate: click handlers, state toggles, dynamically added animation classes. If you relied on JS to add ease-fade-in after mount, the animation was delayed until this step.

What works before vs after hydration

✓ Works immediately (CSS-only)

Entrance classes in static HTML (ease-fade-in, ease-slide-up). Hover classes (ease-hover-grow, ease-hover-lift). Layout utilities (ease-center, ease-flex). Component classes (ease-btn, ease-card).

⚠ Delayed by client:* directive

JS that adds animation classes on mount (useEffect + classList.add). Click-to-animate handlers. State-driven class toggles (isOpen && 'ease-zoom-in'). Counter/toggle components with motion.

✗ Common mistake

Putting ease-fade-in only in JS state instead of static HTML. Using client:visible for above-the-fold hero animations. Expecting hover JS handlers before hydration completes.

client:* directive timing reference

Choose the right hydration strategy based on how critical the interaction is.

Directive When JS hydrates Motion impact Best for
client:load Immediately on page load Minimal delay — JS runs ASAP Critical interactive widgets, modals
client:idle After main thread is idle Small delay — usually unnoticeable Secondary interactions, tooltips
client:visible When element scrolls into view Delayed until scroll — can surprise users Below-fold content, lazy widgets
client:media When media query matches Conditional — mobile vs desktop Responsive-only interactions
client:only Skip SSR, client-only render Blank until hydrated — worst for entrance Browser-only APIs, no SSR needed
(no directive) Never — static HTML only CSS classes work instantly Hero sections, static cards, nav

CSS-only motion 
---
// src/components/Hero.astro
// No client:* directive — pure static HTML
---

<section class="ease-center ease-fade-in">
  <h1 class="ease-slide-up ease-delay-100">
    Welcome
  </h1>
  <button class="ease-btn ease-btn-primary ease-hover-grow ease-delay-200">
    Get Started
  </button>
</section>

Hydration strategy checklist

  • Global CSS: Import easemotion.min.css once in Layout.astro, not per component.
  • Static entrance: Put ease-fade-in / ease-slide-up directly in Astro HTML — not in JS state.
  • CSS hover: Use ease-hover-grow etc. in static markup — CSS :hover works before hydration.
  • Above-fold content: Do not use client:visible or client:only for hero animations.
  • Interactive widgets: Use client:load for modals, toggles, or JS-driven animation triggers.
  • Below-fold lazy: client:visible is fine for off-screen widgets that don't need instant motion.
  • No JS for CSS: If EaseMotion provides a CSS class for the effect, don't recreate it with JS after hydration.
  • Test without JS: Disable JavaScript in DevTools — entrance and hover classes should still work.
