Transform Animation vs Sticky / Fixed Stacking

A parent with ease-fade-in (transform-based entrance) can break a position: sticky child. Scroll both panels — left sticks fail, right works. Classic CSS gotcha, EaseMotion-shaped.

Issue #46194
Why: Transformed ancestors create a new containing block. Sticky / fixed positioning is then resolved against that ancestor — not the viewport — so “sticky stops working” even though the class is fine in isolation.

What happens

  • ease-fade-in uses transform Motion

    Entrance keyframes scale/translate. With animation-fill-mode: both, a transform often remains — enough to create a containing block.

  • Sticky needs an untransformed scrollport chain Gotcha

    If any ancestor between sticky and the scroll container has a transform (or filter/perspective), sticky sticks to that box instead of scrolling with the page/panel as expected.

  • Fixed can be affected too Related

    position: fixed is also containing-block sensitive when ancestors use transform. Same rule: don’t wrap sticky/fixed chrome in entrance-transformed parents.

Do / don’t

  • ✓ Animate content children — leave sticky/fixed ancestors transform-free.
  • ✓ Put ease-fade-in / ease-slide-up on the section body, not the sticky shell.
  • ✓ Prefer opacity-only motion on wrappers that must contain sticky UI.
  • ✕ Don’t put sticky nav/sidebar inside a parent that uses transform entrance classes.
  • ✕ Don’t assume “sticky is broken in EaseMotion” — check containing blocks first.

Copy-paste structures

Don’t Transformed parent 
<!-- ❌ Sticky dies under transformed ancestor -->
<section class="ease-fade-in">
  <aside class="sticky-sidebar">…</aside>
  <div class="content">…</div>
</section>

<!-- ✅ Sticky ancestor stays transform-free -->
