🐛 Incomplete Component Validation List

Issue #40204: Missing 19 Components in validate-pack Checker

The Problem

❌ Current Behavior (Incomplete Validation)

The Issue: The `required` array in `scripts/validate-pack.mjs` contains a hardcoded list of component filenames that must be included in the package.

The Problem: This list is stale and missing 19 newer components that actually exist in the `components/` directory.

The Risk: If any of these 19 components are accidentally omitted from the distribution package, the validator will NOT catch it.

Root Cause Analysis

Why is the List Out of Sync?

1. Hardcoded Component List

The `required` array in `scripts/validate-pack.mjs` is manually maintained with specific component filenames.

// scripts/validate-pack.mjs (Current - Static List) const required = [ 'button.css', 'card.css', 'modal.css', 'tooltip.css', 'badge.css', // ... only ~15 components listed // Missing all 19 newer components! ]

2. Growth Without Synchronization

As new components are added to `components/` directory, the hardcoded list is not automatically updated.

Developers add new files but forget to update the validation script.

3. No Sync Mechanism

There's no automated way to detect when the required list drifts out of sync with the actual file system.

Validation only checks if required components exist, not if ALL components are required.

4. Validation Gap

Current logic:

// ✗ Checks: "Are required components present?" for (const file of required) { if (!packageFiles.includes(file)) fail(); } // ✗ Does NOT check: "Are ALL components present?"

The Solutions

Approach 1: Update Static List

Manual but Safe

✅ Pros

  • Minimal code change
  • Explicit control over requirements
  • No behavior change
  • Easy to review and audit

❌ Cons

  • Manual maintenance required
  • Can drift again in future
  • Error-prone process
  • Requires discipline from developers

Implementation:

const required = [ 'button.css', 'card.css', 'modal.css', 'tooltip.css', 'badge.css', // ... existing components // Add missing 19 components: 'announce-bar.css', 'avatar.css', 'breadcrumb.css', 'btn-magnetic.css', 'command-palette.css', 'compare-table.css', 'connection-status.css', 'fab.css', 'forms.css', 'kbd.css', 'pagination.css', 'password-strength.css', 'progress.css', 'read-more.css', 'scroll-gallery.css', 'skeleton.css', 'tag.css', 'toast.css', 'view-transitions.css' ];

Approach 2: Dynamic Discovery

Automatic and Future-Proof

✅ Pros

  • Auto-discovers new components
  • Never drifts out of sync
  • Future-proof solution
  • Less maintenance burden

❌ Cons

  • More complex implementation
  • Requires file system scan
  • May include unwanted files
  • Need clear inclusion logic

Implementation:

const fs = require('fs'); const path = require('path'); /** * Dynamically discover all component CSS files * Ensures no components are missed as new ones are added */ function getRequiredComponents() { const componentsDir = path.join(__dirname, '../components'); try { const files = fs.readdirSync(componentsDir) .filter(file => file.endsWith('.css')) .sort(); console.log(`Found ${files.length} components`); return files; } catch (err) { console.error('Failed to scan components:', err); process.exit(1); } } const required = getRequiredComponents(); // Now always in sync with actual components!

🎯 Recommended Solution

Use Approach 2 (Dynamic Discovery) as the primary solution because:

  • ✅ Automatically includes all present and future components
  • ✅ Prevents this issue from recurring
  • ✅ Reduces maintenance burden on developers
  • ✅ Follows DRY principle (Don't Repeat Yourself)
  • ✅ Simplifies onboarding for new component contributors

As a fallback: If dynamic discovery is not feasible, implement Approach 1 (update static list) immediately and establish a process to keep it synchronized.
