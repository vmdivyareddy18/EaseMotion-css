Astro Islands & EaseMotion CSS

Understand how client:* hydration timing affects EaseMotion hover and entrance animations — and how to keep motion smooth and predictable.

Concepts Directives Pitfalls Patterns Chooser Snippets

Core Concepts

Before diving into problems and solutions, let's align on what each term means in this context.

🏝️

Astro Islands

Interactive UI components rendered inside an otherwise static Astro page. Each island is an isolated JavaScript bundle that hydrates independently.

💧

Hydration

The process where the browser downloads a component's JS, runs it, and makes it interactive. Until hydration completes, the component is a static HTML snapshot.

🎨

EaseMotion CSS

A utility CSS framework. Because it is pure CSS, animations like entrance fades and hover effects are always active — they require no JavaScript to run.

⚡

The Mismatch

CSS animations run immediately. JS-driven interactions (click handlers, state changes) wait for hydration. Confusing the two is the source of most Astro + motion bugs.

💡
Key insight: EaseMotion entrance and hover animations work in any Astro component — hydrated or not — because they are driven entirely by CSS. Hydration only matters when you need JavaScript to add or remove an EaseMotion class at runtime.

Hydration Directives & Timing

Each client:* directive decides when a component becomes interactive. Click a directive to see its timing and motion impact.

Fastest client:load ▼

When: Hydrates immediately as the page loads — highest priority.

Motion impact: Minimal delay. The component is interactive almost as soon as the HTML paints. Safe for hero animations and above-the-fold entrance effects that also need JS class toggling.

Hydrates: immediately on page load

Deferred client:idle ▼

When: Waits for the browser's requestIdleCallback — after high-priority work finishes.

Motion impact: Can be 200 ms – 2 s after paint on busy pages. CSS entrance animations will already be done. JS-toggled hover classes will silently fail until hydration completes — users may click with no visible response.

Hydrates: after browser idle (variable delay)

Lazy client:visible ▼

When: Uses IntersectionObserver — hydrates only when the component enters the viewport.

Motion impact: Perfect for scroll-triggered CSS entrance animations (they fire on scroll naturally). Dangerous for hover effects: the user can hover before hydration, triggering the CSS animation but not any JS handler.

Hydrates: when scrolled into viewport

Conditional client:media ▼

When: Hydrates when a CSS media query matches (e.g. client:media="(max-width: 768px)").

Motion impact: Component is static on non-matching viewports. EaseMotion CSS animations still run. JS-dependent interactions are dead until the breakpoint is reached.

Hydrates: when media query matches

No JS client:only ▼

When: Renders entirely client-side — no SSR HTML. Hydrates on load like client:load but skips server rendering.

Motion impact: There is no server HTML snapshot, so entrance animations play after the JS bundle downloads and renders the component. This can cause a blank flash before the animation starts on slow connections.

Hydrates: client-only render (no SSR)

Common Pitfalls

These are the patterns that look right but silently break motion UX in Astro projects. Click each card to expand the explanation.

01

JS toggling an EaseMotion class before hydration

▼
❌ Problematic
// Button.jsx — uses client:idle
const [active, setActive] = useState(false);
return (
  <button
    className={`btn ${active ? "ease-pulse" : ""}`}
    onClick={() => setActive(true)}
  >
    Click me
  </button>
);

User clicks during idle delay → no animation, no feedback. Hydration hasn't run yet so onClick is dead.

✅ Fix: use client:load
// Switch to client:load in the .astro file
<Button client:load />

// Or keep client:idle but add CSS-only fallback
<button
  class="btn ease-hover-scale"
  data-js-pulse
>
  Click me
</button>
/* CSS hover works always.
   JS adds ease-pulse only after hydration. */

CSS hover animation is always live. JS enhancement layers on top after hydration.

02

Entrance animation replays after client:only renders

▼
❌ Problematic
// Card.astro — hero card
<Card client:only="react" />

// Card.jsx
return <div className="card ease-fade-in">...</div>;

No SSR HTML → blank space until JS loads → card pops in and plays fade-in. Users on slow connections see a layout jump.

✅ Fix: use client:visible or SSR
// Prefer client:visible for below-fold cards
<Card client:visible />

// Or extract static shell to .astro,
// keep only interactive part as island
<div class="card ease-fade-in">
  <!-- static SSR content -->
  <CardInteractive client:idle />
</div>

Static shell renders server-side, so CSS entrance runs immediately. Only the interactive part waits for JS.

03

Scroll entrance fires before client:visible hydrates

▼
❌ Problematic
// Section.astro
<AnimatedSection client:visible />

// AnimatedSection.jsx — adds class on scroll
useEffect(() => {
  const obs = new IntersectionObserver(([e]) => {
    if (e.isIntersecting)
      ref.current.classList.add("ease-slide-up");
  });
  obs.observe(ref.current);
}, []);

The CSS entrance plays when scrolled into view (CSS is always live). Then hydration fires and re-adds the class via JS — animation replays unexpectedly.

✅ Fix: check if class already present
useEffect(() => {
  const obs = new IntersectionObserver(([e]) => {
    if (e.isIntersecting
      && !ref.current.classList
           .contains("ease-slide-up")) {
      ref.current.classList.add("ease-slide-up");
    }
  });
  obs.observe(ref.current);
}, []);

Guard prevents double-firing. Or use a CSS-only scroll reveal so JS is never needed.

Recommended Patterns

Four integration patterns that keep motion smooth regardless of hydration timing.

A

CSS-first, JS-enhanced

Always write CSS-driven motion as the baseline. Layer JS class toggling on top only where needed. Users get smooth animation before hydration; JS adds richer interaction after.

<!-- Always has hover animation via CSS -->
<button class="btn ease-hover-lift"
        data-ripple>
  Submit
</button>
<!-- JS adds ripple AFTER hydration -->
B

Static shell + minimal island

Keep the visual structure in .astro with EaseMotion classes applied at build time. Only wrap the smallest interactive piece in a framework component.

<!-- Page.astro -->
<section class="ease-fade-in">
  <h2>Static title</h2>
  <p>Static body</p>
  <LikeButton client:idle />
</section>
C

is-hydrated guard class

Add a sentinel class once hydration completes. Use it in CSS to enable JS-dependent animations, preventing them from triggering before the component is interactive.

// Component.jsx
useEffect(() => {
  ref.current.classList
    .add('is-hydrated');
}, []);

/* style.css */
.card .ease-pulse { display: none; }
.card.is-hydrated .ease-pulse {
  display: block;
}
D

animation-fill-mode: both

For entrance animations on client:only components, add animation-fill-mode: both and a small delay. This hides the blank-flash period and lets the animation play cleanly once the component mounts.

/* style.css */
.island-entrance {
  animation: ease-kf-fade-in 0.5s
    ease-out 0.1s both;
  /* 'both' = stays at keyframe 0
     until the delay passes */
}

Directive Chooser

Answer two questions and get the recommended hydration directive for your use case.

Does the component need JavaScript to function correctly on first interaction?

Is this component above the fold / in the hero area?

Does the component appear in the viewport on load?

client:load

Critical above-fold interactivity — hydrate immediately. Pair EaseMotion entrance classes directly; they will play as the page paints.

client:idle

Non-critical interactivity — hydrate after load. Use pattern A (CSS-first) so hover and entrance animations work before the JS kicks in.

client:visible

Off-screen component — hydrate on scroll. Entrance animations are safe. Add the double-fire guard (Pattern C) if JS also adds motion classes.

No directive needed

EaseMotion CSS classes work with zero JavaScript. Keep the component as a plain .astro file or static HTML — no island required.

CSS-first hover + JS enhancement (React island) 
// Button.jsx — used with client:idle
import { useRef, useEffect } from 'react';

export default function Button({ label }) {
  const ref = useRef(null);

  // Adds ripple class only after hydration
  useEffect(() => {
    ref.current.addEventListener('click', () => {
      ref.current.classList.add('ease-pulse');
      setTimeout(() =>
        ref.current.classList.remove('ease-pulse'), 600);
    });
  }, []);

  return (
    <button
      ref={ref}
      // ease-hover-lift works BEFORE hydration via CSS
      className="btn ease-hover-lift"
    >
      {label}
    </button>
  );
}
is-hydrated guard for JS-gated animations 
// AnimatedWidget.jsx — client:visible
import { useRef, useEffect } from 'react';

export default function AnimatedWidget() {
  const ref = useRef(null);

  useEffect(() => {
    // Signal that JS is ready
    ref.current?.classList.add('is-hydrated');
  }, []);

  return <div ref={ref} className="widget">...</div>;
}

/* style.css */
/* JS-only animation hidden until hydrated */
.widget .js-only-pulse  { display: none; }
.widget.is-hydrated .js-only-pulse { display: block; }
client:only blank-flash fix with animation delay 
// Page.astro
<HeavyWidget client:only="react" />

/* HeavyWidget.module.css — applied in the component */
.root {
  /* Stays invisible at keyframe-0 during JS load delay */
  animation: ease-kf-fade-in 0.45s ease-out 0.15s both;
}

// HeavyWidget.jsx
import styles from './HeavyWidget.module.css';
export default function HeavyWidget() {
  return <div className={styles.root}>...</div>;
}
