Global Reduced-Motion vs Engine Class Double-Apply

Why DevTools shows two @media (prefers-reduced-motion: reduce) blocks on the same element — and why that is intentional, not a bug.

Short answer: Both rules apply the same outcome (animation-duration: 0.01ms !important). The global rule in easemotion.css covers everything; the engine's per-class guard exists so em="" animations still respect reduced motion when the full CSS bundle is not loaded. Neither "wins" over the other — they reinforce the same result.

Who wins?

When reduced motion is active, the global * rule and the engine's ._em_* rule both set animation-duration: 0.01ms !important. DevTools may show one as struck through due to equal specificity + !important, but the computed value is identical. There is no conflict — only redundancy.

Global rule vs engine per-class guard

Property Global (easemotion.css) Engine (compiler.js)
Selector *, *::before, *::after ._em_{hash} (specific class)
animation-duration 0.01ms !important 0.01ms !important
animation-iteration-count 1 !important 1 !important
transition-duration 0.01ms !important not set
scroll-behavior auto !important not set
When needed Always (full bundle loaded) Engine standalone / em="" without full CSS
Source file easemotion.css (end of bundle) easemotion/engine/compiler.js

Why both rules exist

Global rule purpose

The catch-all * selector at the end of easemotion.css ensures every animation, transition, and scroll behavior respects the user's OS-level reduced-motion setting — framework classes, components, and utilities.

Engine guard purpose

The Motion Engine can run standalone (importing only easemotion-css/engine without the full CSS bundle). Per-class guards ensure em="" injected animations still respect reduced motion even when the global rule is absent.

DevTools confusion

When both are loaded, DevTools lists two @media blocks. One may appear struck through. This is normal cascade display — not a bug or override conflict. The computed animation duration is 0.01ms either way.

Actual source snippets

From the real framework files — not simplified.

Global rule — easemotion.css

/* easemotion.css — end of bundle */
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

Engine per-class guard — compiler.js

/* easemotion/engine/compiler.js — compile() output */
._em_a3f2c1 {
  animation: ease-kf-fade-in 500ms cubic-bezier(0,0,0.2,1) both;
}
@media (prefers-reduced-motion: reduce) {
  ._em_a3f2c1 {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
  }
}

How to test reduced motion locally

/* Option 1 — OS setting */
Windows: Settings → Accessibility → Visual effects → Animation effects OFF
macOS: System Settings → Accessibility → Display → Reduce motion ON

/* Option 2 — Chrome DevTools */
1. Open DevTools (F12)
2. Cmd/Ctrl + Shift + P → type "Rendering"
3. Open Rendering tab → Emulate CSS media feature
4. Select prefers-reduced-motion: reduce

/* Option 3 — Firefox */
about:config → ui.prefersReducedMotion → 1

/* Verify in DevTools Computed tab */
animation-duration should show 0.01ms on any .ease-* element
