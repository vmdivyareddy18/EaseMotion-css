🐛 Repository URL Validation Bug Fix

Issue #40054: Case-Sensitive URL Validation in validate:manifest

The Problem

❌ Current Behavior (Fails)

package.json repository URL:
"url": "git+https://github.com/saptarshi-coder/easemotion-css.git"

validate-package.mjs expects:
"SAPTARSHI-coder/EaseMotion-css"

Result: ✗ Validation Failed
Error: "repository.url must point to SAPTARSHI-coder/EaseMotion-css"

⚠️ Impact Chain

npm run validate:manifest
❌ BLOCKED
↓
release:check
❌ BLOCKED
(cannot proceed to bundle, lint, test checks)

The Solution

✅ Normalize GitHub URLs Case-Insensitively

Update scripts/validate-package.mjs to normalize GitHub owner/repo names before comparison:

❌ Before (Case-Sensitive)

const expectedUrl = 'git+https://github.com/SAPTARSHI-coder/EaseMotion-css.git'; if (pkg.repository.url !== expectedUrl) { throw new Error( `repository.url must point to SAPTARSHI-coder/EaseMotion-css` ); }

✅ After (Case-Insensitive)

const normalizeGitHubUrl = (url) => { return url.toLowerCase() .replace(/\.git$/, ''); }; const expectedPattern = 'https://github.com/saptarshi-coder/easemotion-css'; const normalizedUrl = normalizeGitHubUrl(pkg.repository.url); if (!normalizedUrl.includes(expectedPattern)) { throw new Error( `repository.url must point to saptarshi-coder/easemotion-css (any casing)` ); }

Implementation Steps:

  1. Add a normalizeGitHubUrl() helper function
  2. Convert both expected URL and actual URL to lowercase
  3. Remove .git suffix for comparison
  4. Use case-insensitive comparison
  5. Test with npm run validate:manifest

Steps to Reproduce the Bug

# In the EaseMotion CSS repository root:

$ npm ci
$ npm run validate:manifest

Error: package.json validation failed:
repository.url must point to SAPTARSHI-coder/EaseMotion-css

Technical Details

Root Cause

The validator performs a strict string comparison without normalizing GitHub repository URLs, which differ in casing conventions (GitHub URLs are lowercase, but project branding may use mixed case).

Why It Matters

GitHub URLs are case-insensitive (github.com/User/Repo resolves to lowercase), but the validator expected exact casing, causing validation to fail on checked-in code.

Best Practice

Always normalize external identifiers (URLs, usernames, etc.) before comparison to avoid case-sensitivity issues across different systems.

Testing Strategy

Test with multiple URL casing variations to ensure the validator accepts all valid GitHub URL formats.

Valid URL Examples (All Should Pass)
