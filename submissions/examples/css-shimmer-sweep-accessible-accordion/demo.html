Installation Add AccessAccord to any HTML page in one step
Stable +

AccessAccord is a standalone CSS file. Include EaseMotion CSS for page-level animations, then add the component stylesheet. No npm, no bundler, no JavaScript runtime required.

<!-- Step 1: EaseMotion CSS --> <link rel="stylesheet" href="easemotion.css"> <!-- Step 2: AccessAccord --> <link rel="stylesheet" href="access-accord.css"> <!-- Step 3: Use it --> <div class="accordion"> <details class="accordion-item"> <summary class="acc-header">...</summary> <div class="acc-panel">...</div> </details> </div>
✓Works in every modern browser without polyfills
✓Zero JavaScript — no parser, no event listeners, no runtime
✓Single CSS file — approximately 6 KB minified and gzipped
✓Compatible with any HTML framework, CMS, or static site generator

Keyboard Navigation Complete keyboard control with no custom scripting
Stable +

Because AccessAccord is built on native <details> and <summary> HTML elements, the browser provides complete keyboard accessibility without a single line of JavaScript.

Tab Move focus to the next accordion header (or interactive element)
Shift+Tab Move focus to the previous accordion header
Enter Toggle the focused accordion panel open or closed
Space Toggle the focused accordion panel open or closed
Tab (inside panel) Move through any links or interactive elements within the open panel

When focus is on a header, a high-visibility 3px blue outline appears — meeting WCAG 2.1 Success Criterion 2.4.11 (Focus Appearance) which requires a focus indicator with minimum area and contrast.

🔊
Screen Reader Support Native semantics — no ARIA hacks required
Stable +

The <details> element has a native disclosure widget role in the accessibility tree. Screen readers announce each <summary> as a button with an "expanded" or "collapsed" state — exactly what the ARIA pattern requires — without any role, aria-expanded, or aria-controls attributes needed.

✓NVDA + Chrome: announces "Summary text, collapsed, button" and "expanded" on open
✓JAWS + Chrome/Edge: reads header text and state correctly
✓VoiceOver + Safari (macOS/iOS): "disclosure triangle, collapsed/expanded"
✓TalkBack + Android Chrome: correct state and interaction
✓Decorative elements (chevron, icon) carry aria-hidden="true" to suppress noise

Panel content is only in the accessibility tree when the item is open, so screen reader users can efficiently navigate without wading through hidden content.

🎯
Focus Management Visible, high-contrast focus indicators on every element
Stable +

Every interactive element in AccessAccord has a clearly visible focus ring styled via :focus-visible. This ensures keyboard users always know where focus is, while keeping the interface clean for pointer users who don't need the ring.

Element Focus Style WCAG Criterion
Accordion header (summary) 3px solid blue, inset offset SC 2.4.7, SC 2.4.11
Nav links 3px solid blue, 2px offset SC 2.4.7
CTA buttons 3px solid blue, 3px offset SC 2.4.7
Footer links 2px solid blue-light, 2px offset SC 2.4.7
Social icons 3px solid blue-light, 2px offset SC 2.4.7

The --accordion-focus CSS variable controls the focus ring colour globally, making it easy to match your brand while maintaining compliance.

/* Override the focus ring colour */ .my-section { --accordion-focus: #0891b2; /* cyan-600 */ }

Customization

Five CSS Variables

Override any variable on an ancestor element. Changes cascade to all accordion items within that scope — no source edits required.

// Property Reference
Accordion CSS custom properties, their defaults, and what they control
--accordion-radius
Default: 10px
Border radius on each accordion card. Reduce to 4px for a sharp government-style look; increase to 16px for a softer product feel.
--accordion-duration
Default: 0.36s
Base timing for the shimmer sweep and all internal transitions. The shimmer runs at 2.5× this value for a naturally flowing sweep.
--accordion-highlight
Default: rgba(219,234,254,0.82)
The colour and opacity of the shimmer sweep highlight layer. Use a brand tint or neutral white for different aesthetics.
--accordion-focus
Default: #2563eb
Focus ring colour applied to all interactive elements. Always choose a value with ≥ 3:1 contrast against adjacent colours (WCAG SC 1.4.11).
--accordion-border
Default: #e2e8f0
Default card border colour. Increasing contrast here improves component legibility in bright environments or for low-vision users.
// Override Examples
/* High-contrast government theme */
.gov-section {
  --accordion-radius:    4px;
  --accordion-focus:     #ffdd00;
  --accordion-highlight: rgba(255, 221, 0, 0.2);
  --accordion-border:    #0b0c0c;
}
/* Teal brand theme */
.brand-teal {
  --accordion-focus:     #0d9488;
  --accordion-highlight: rgba(204, 251, 241, 0.8);
  --accordion-duration:  0.28s;
}
/* Slow, prominent shimmer */
.marketing-page {
  --accordion-duration:  0.8s;
  --accordion-highlight: rgba(255, 255, 255, 0.9);
  --accordion-radius:    14px;
}
All overrides work on any ancestor element — scope them to a single section or the entire page.
Best Practices

Implementing Responsibly

Eight guidelines for using accessible accordions on public-facing websites — informed by WCAG 2.1, GOV.UK design patterns, and the ARIA Authoring Practices Guide.

1
Only use accordions when content can be skipped
Accordions are best when users may not need all the information — FAQs, settings panels, long documentation. Don't hide content that every user needs to read, such as critical instructions or mandatory disclosures.
2
Use descriptive, unique header text
Each <summary> must make sense out of context — screen reader users often navigate by listing all interactive elements. Avoid vague labels like "Click here" or "More". Use specific labels like "Keyboard Navigation Guide".
3
Don't auto-expand all panels on load
Opening all panels defeats the purpose of the accordion pattern and forces screen reader users to wade through all content sequentially. Let users choose what to open. Use the open attribute sparingly — only when one panel is clearly most relevant.
4
Verify contrast with automated tools
Before deploying with custom colour overrides, verify all text and UI components using a contrast checker (axe DevTools, Colour Contrast Analyser, or browser devtools). Target a 4.5:1 ratio for normal text and 3:1 for large text and UI components.
5
Test with real assistive technology
Automated accessibility testing catches around 30–40% of issues. Supplement with manual testing using NVDA + Chrome (Windows), VoiceOver + Safari (macOS/iOS), and TalkBack + Chrome (Android) to verify real-world screen reader behaviour.
6
Provide visible skip links for long pages
When using multiple accordions on a long page, include a "Skip to main content" link as the first focusable element. This page includes one — press Tab with focus at the browser address bar to see it appear.
7
Keep panel content scannable
Once a panel is open, it becomes part of the document flow. Use short paragraphs, bullet lists, and descriptive subheadings inside panels so keyboard and screen reader users can navigate through the content efficiently without reading everything.
8
Respect user animation preferences
The shimmer sweep and panel open animations are suppressed automatically when prefers-reduced-motion: reduce is active. If you add any additional animations, always include a corresponding reduced-motion override to avoid triggering vestibular disorders.
