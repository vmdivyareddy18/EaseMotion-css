Scroll-Driven Animations: The Future of CSS

Scroll-driven animations are a new CSS capability that links animation progress directly to the scroll position of a container — no JavaScript, no IntersectionObserver, no requestAnimationFrame. The browser handles all the heavy lifting.

The animation-timeline property

The key is the animation-timeline property. Setting it to scroll(root block) means the animation progresses as the root element (the viewport) scrolls vertically. Combined with animation-range, you control exactly when within the scroll range the animation plays.

"CSS scroll-driven animations give us precisely what JS-based scroll handlers have been doing for years — but declared in CSS, with zero JavaScript overhead."

Reading progress bars

A progress bar showing how far through an article the user has read is the most natural use case. The bar's scale transforms from 0 to 1 as the user scrolls from the top to the bottom of the page. Using transform-origin: left and scaleX is more performant than animating width, since scale changes are GPU-composited.

Section-level progress

By using scroll(nearest block) instead of scroll(root block), you can track progress within a specific scrollable container rather than the whole page. This is useful for multi-step wizards, carousels, or tabs with their own scroll areas.

Percentage labels

The percentage badge in the top-right corner also uses a scroll-driven animation — its opacity fades from 0 to 1 between the 5% and 100% scroll range (it hides when you're at the very top). The actual number displayed is updated by a small JS snippet, since CSS alone cannot format a dynamic percentage string.

Browser support

Scroll-driven animations landed in Chrome 115+ and Edge 115+. Firefox support arrived in version 110 with the layout.css.scroll-driven-animations.enabled flag, and is enabled by default in Firefox 126+. Safari is still working on implementation. The reduced-motion fallback in this demo uses a CSS custom property driven by a JS scroll listener.

Performance

Because scroll-driven animations run on the compositor thread (when they only animate compositable properties like transform and opacity), they remain smooth at 60fps or 120fps even on throttled CPUs. This makes them strictly superior to JavaScript scroll handlers for these effects.
