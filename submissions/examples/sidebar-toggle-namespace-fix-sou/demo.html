Sidebar Toggle — Namespace Pollution Fix

Issue #2482 — the generic .sidebar-toggle selector collides with application-level navigation styles. The fix is to use the framework- prefixed .ease-sidebar-toggle selector (already present in components/sidebar.css).

Below, a fictional host app defines its own .sidebar-toggle { background: red; … } rule before EaseMotion loads. Watch what happens.

🔴 Before — buggy

Unprefixed .sidebar-toggle

What happened: the host application also defined a .sidebar-toggle rule. CSS specificity + load order make the host app's red styling win, breaking the framework's intended appearance. This is the namespace pollution described in issue #2482.

<!-- markup --> <button class="sidebar-toggle">☰ Toggle</button> /* host-app.css (loaded by the user) */ .sidebar-toggle { background: red; … } /* 💥 collides */
🟢 After — fixed

Prefixed .ease-sidebar-toggle

What changed: the framework selector is namespaced with ease-. No host-application rule named .sidebar-toggle can ever override it, because the selectors don't match. The collision is impossible by construction.

<!-- markup --> <button class="ease-sidebar-toggle">☰ Toggle</button> /* host-app.css (loaded by the user) */ .sidebar-toggle { background: red; … } /* ✅ harmless */

Bonus — collapsed-state rotation still works

Click the button to toggle the ease-sidebar-collapsed modifier on its parent layout. The framework's existing rotation behavior in components/sidebar.css continues to work with the prefixed selector — proving the fix is fully backwards compatible.

📋 Migration checklist for users

  1. Search your project for class="sidebar-toggle".
  2. Replace with class="ease-sidebar-toggle".
  3. If you have JS selectors like document.querySelector('.sidebar-toggle'), update them too.
  4. Re-test your sidebar — appearance and behavior should be identical, but now collision-proof.
