Reading Flow CSS Utilities

A utility toolkit for the experimental reading-flow CSS property — controlling how keyboard focus and screen readers traverse flex and grid layouts when visual order diverges from DOM order.

CSS-only Accessibility Experimental
⚠
Experimental Property

reading-flow is currently available behind a flag in Chrome 125+ (enable Experimental Web Platform Features at chrome://flags) and is not yet supported in Firefox or Safari. The spec is still under active development in the CSS Working Group. Use in production only with a suitable progressive-enhancement fallback. Tab order falls back to DOM order in unsupported browsers.

MDN reference →

What is reading-flow?

CSS layout tools like flexbox and grid let us reorder elements visually via order, flex-direction: row-reverse, or placement properties. But keyboard focus still follows DOM order — creating a disconnect for keyboard-only and assistive-technology users.

The reading-flow property instructs the browser to walk focus (and reading order) through a flex or grid container in the order items appear on screen, rather than their source order. These utilities wrap each value into a clean, composable class.

🗂

Source vs Visual

DOM order controls keyboard nav by default. Visual reordering breaks that contract.

🔑

reading-flow fixes it

Aligns focus traversal with what the user actually sees, per the chosen strategy.

🧩

Six strategies

Different values handle flex direction, flex flow, grid rows, columns, and CSS order.

.reading-flow-normal

value: normal

Resets any inherited reading-flow to its default. Focus walks in DOM source order — the browser's standard behaviour. Useful to opt a nested container back out of a parent's reading-flow strategy. 

/* Reset to browser default — DOM source order */
.reading-flow-normal {
  reading-flow: normal;
}
Live Example — tab through to see focus order

Focus moves 1 → 2 → 3 → 4 → 5 — same as DOM order. No reordering applied here, so normal is the correct value.

.reading-flow-flex-visual

value: flex-visual

Inside a flex container, focus traverses items in their visual (rendered) position — matching the direction set by flex-direction and writing mode. Ideal for reversed or wrapped flex rows where items shift visually but not in the DOM. 

/* Follow visual flex position, respecting writing-mode */
.reading-flow-flex-visual {
  reading-flow: flex-visual;
}
Example — flex-direction: row-reverse

Items are ordered 5 4 3 2 1 in the DOM, but rendered right-to-left. Without reading-flow, tab order would go 5→4→3→2→1 (DOM). With flex-visual, it correctly follows the screen: 1→2→3→4→5.

Tab from left → right: focus should read 1 → 2 → 3 → 4 → 5 (visual order), not 5 → 4 → 3 → 2 → 1 (DOM order). 

<div class="flex reading-flow-flex-visual" style="flex-direction: row-reverse">
  <button>5</button>
  <button>4</button>
  <button>3</button>
  <button>2</button>
  <button>1</button>
</div>

.reading-flow-flex-flow

value: flex-flow

Similar to flex-visual, but ignores the writing mode direction. Focus walks in the flex-flow axis — useful for containers where you explicitly want consistent left-to-right or top-to-bottom traversal regardless of the document's writing direction. 

/* Follow flex flow axis — ignores writing-mode direction */
.reading-flow-flex-flow {
  reading-flow: flex-flow;
}
Example — column flex with out-of-order DOM

Items flow top-to-bottom in a column. flex-flow ensures focus follows that visual column order rather than any CSS order tweaks.

DOM order: A, B, C. CSS order: A=1, C=2, B=3. Focus should visit A → C → B (visual top-to-bottom). 

<div class="flex reading-flow-flex-flow" style="flex-direction: column">
  <button style="order: 1">A</button>
  <button style="order: 3">B</button>
  <button style="order: 2">C</button>
</div>

.reading-flow-grid-rows

value: grid-rows

For grid containers, traverses focusable items row by row — left to right within each row, then down to the next. This matches the natural reading pattern for most Western layouts, even when grid placement puts items out of source order. 

/* Traverse grid row by row, left→right, top→bottom */
.reading-flow-grid-rows {
  reading-flow: grid-rows;
}
Example — grid with explicit placement

Items are placed with grid-column / grid-row. Source order doesn't match visual position. grid-rows makes focus sweep each row correctly.

DOM order: B, F, A, D, C, E. Visual row order: A → B → C → D → E → F. Focus follows visual rows.

.reading-flow-grid-columns

value: grid-columns

Traverses grid items column by column — top to bottom within each column, then to the next column. This suits vertical-priority designs like magazine layouts or masonry-style grids. 

/* Traverse grid column by column, top→bottom, left→right */
.reading-flow-grid-columns {
  reading-flow: grid-columns;
}
Example — same grid, column-first focus

Using the same 3×2 grid from above, but now focus walks A → D → B → E → C → F (column by column).

Column-first order: A → D (col 1), B → E (col 2), C → F (col 3).

.reading-flow-grid-order

value: grid-order

Respects the CSS order property on grid items — focus visits items in the order you've explicitly set via order, combining CSS order with grid placement for full control over traversal sequence. 

/* Follow CSS `order` property values within the grid */
.reading-flow-grid-order {
  reading-flow: grid-order;
}
Example — grid items with explicit CSS order

Items sit in a grid, but each carries an order value that defines the desired focus sequence, regardless of visual position.

Focus follows the order values: ① → ② → ③ → ④ → ⑤ → ⑥, regardless of which grid cell each item occupies. 

<div class="grid reading-flow-grid-order">
  <button style="grid-column:3; order:1">First focus</button>
  <button style="grid-column:2; order:2">Second focus</button>
  <button style="grid-column:1; order:3">Third focus</button>
</div>

Accessibility Notes

🎯

Match reading order

Only use reading-flow values that genuinely match how a human reader would expect to move through the content. Arbitrary orders are confusing for everyone.

🔭

Test with real AT

Screen reader virtual cursors may not respect reading-flow in the same way keyboard focus does. Always test with NVDA, JAWS, and VoiceOver.

🧱

Don't replace ARIA

reading-flow adjusts focus order, not semantics. Use appropriate ARIA roles and landmarks alongside it, not instead of them.

📐

Progressive enhancement

Browsers that don't support reading-flow fall back to DOM order. Ensure DOM order is still a logical fallback — don't rely solely on CSS for a meaningful experience.

🎨

Visible focus styles

These demos include high-contrast focus rings. Your production layout should maintain at least a 3:1 contrast ratio for focus indicators (WCAG 2.2 §2.4.11).

↩

Avoid tabindex hacks

reading-flow is designed to replace tabindex juggling. Using both simultaneously leads to unpredictable behaviour.
